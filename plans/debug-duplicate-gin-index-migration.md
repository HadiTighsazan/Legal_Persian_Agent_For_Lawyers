# Root Cause Analysis: `relation "chunk_search_vector_gin" already exists`

## Problem

Running `docker-compose exec backend pytest -v` causes 75 test errors, all with:

```
django.db.utils.ProgrammingError: relation "chunk_search_vector_gin" already exists
```

Every test in the `conversations/` app fails during test database setup because Django's test runner runs all pending migrations, and migration `0017` tries to create a GIN index that already exists.

## Root Cause

The GIN index `chunk_search_vector_gin` on `document_chunks.search_vector` is being created **three times** across the migration chain:

| Migration | How | Status |
|-----------|-----|--------|
| `0006_add_fts_and_metadata_fields.py` (line 135) | `RunSQL` with `CREATE INDEX IF NOT EXISTS` | ✅ Safe — uses `IF NOT EXISTS` |
| `0014_documentchunk_chunk_search_vector_gin.py` | `RunSQL` with `CREATE INDEX IF NOT EXISTS` | ✅ Safe — uses `IF NOT EXISTS` |
| `0017_documentchunk_chunk_search_vector_gin.py` | Django `AddIndex` operation | ❌ **FAILS** — Django's `AddIndex` does **not** use `IF NOT EXISTS` |

### Why it happens

1. **Migration `0006`** (original) created the index via `RunSQL` with `CREATE INDEX IF NOT EXISTS`.
2. **Migration `0014`** was added later (for a different branch/feature) — also used `RunSQL` with `IF NOT EXISTS`, so it was safe.
3. **Migration `0017`** was auto-generated by `makemigrations` after the `GinIndex` was added to the model's `Meta.indexes` in [`documents/models.py:180`](src/backend/documents/models.py:180). This uses Django's built-in [`AddIndex`](https://docs.djangoproject.com/en/4.2/ref/migration-operations/#addindex) operation, which translates to `CREATE INDEX ...` **without** `IF NOT EXISTS`.

When the test runner creates the test database, it runs all migrations sequentially:
- `0006` creates the index (safe)
- `0014` tries to create it again but `IF NOT EXISTS` prevents error (safe)
- `0017` tries to create it a third time via `AddIndex` → **CRASH** because the index already exists

### Why `--reuse-db` doesn't help

The `pytest.ini` has `--reuse-db`, but this only reuses the test database between runs. On the **first** run (or when the test DB is recreated), all migrations run fresh, hitting the error.

## The Fix

### Option A (Recommended): Change migration `0017` to use `RunSQL` with `IF NOT EXISTS`

Replace the `AddIndex` operation in [`0017_documentchunk_chunk_search_vector_gin.py`](src/backend/documents/migrations/0017_documentchunk_chunk_search_vector_gin.py) with a `RunSQL` operation using `CREATE INDEX IF NOT EXISTS`, matching the pattern used in `0006` and `0014`.

This is the safest fix because:
- It's consistent with the two prior migrations that created the same index
- `IF NOT EXISTS` is idempotent — safe for both fresh and existing databases
- No need to `--fake` any migrations

### Option B: Remove migration `0017` entirely

Since the index already exists from `0006` and `0014`, migration `0017` is redundant. However, this would require:
- Deleting the migration file
- Removing the `GinIndex` from `Meta.indexes` in the model (or accepting a perpetual "unapplied migration" warning)
- This is more invasive and error-prone
