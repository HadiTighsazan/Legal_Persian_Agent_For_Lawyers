# Diagnostic Plan: Irrelevant Source Retrieval for Military Service Query

## Problem Statement

When the user asked:  
**"پسری که مشمول باشه ولی خودشو معرفی نکنه جرمش فرار از خدمت است؟"**  
(Is it a crime of desertion if a conscript doesn't report for military service?)

The system returned sources from:
- **Civil Code material 1210** (about guardianship/حجر و قیمومت) — completely irrelevant
- **Material 504** (about inciting military personnel to desertion) — tangentially related but not the right law

The user confirms they **have** the relevant data (military service laws like "قانون مجازات جرایم نیروهای مسلح") in the system, but the RAG pipeline failed to retrieve them.

## Architecture Summary

The Global RAG pipeline for this endpoint (`/legal-research/{conversation_id}`) follows this flow:

```
User Query
    │
    ▼
┌──────────────────────────────┐
│ 1. Question Router (LLM)     │ ← route_question()
│    Determines relevant hubs  │
│    Generates per-hub queries │
└──────────┬───────────────────┘
           │ sub_queries: {hub: {fts_query, vector_query}}
           ▼
┌──────────────────────────────┐
│ 2. Multi-Hub Search          │ ← multi_hub_search()
│    Per hub:                  │
│    a. Embed vector_query     │
│    b. cross_document_hybrid  │
│       _search(hub_type, ...) │
│       - Vector (pgvector)    │
│       - Keyword (FTS)        │
│       - Trigram (pg_trgm)    │
│       - RRF fusion [3,1,1]   │
└──────────┬───────────────────┘
           │ hub_results: {hub: [chunks]}
           ▼
┌──────────────────────────────┐
│ 3. Per-Hub Partial Answers   │ ← generate_hub_partial_answer()
│    3 LLM calls (1 per hub)   │
└──────────┬───────────────────┘
           │ partial_answers
           ▼
┌──────────────────────────────┐
│ 4. Answer Synthesis          │ ← synthesize_answers()
│    1 LLM call with conflict  │
│    detection                 │
└──────────┬───────────────────┘
           ▼
    Final Answer + Sources
```

## Potential Failure Points (Hypotheses)

### Hypothesis A: Question Router Failure
The LLM-based question router might have:
- Incorrectly determined that "legislation" hub is NOT relevant for this query
- Generated poor sub-queries (e.g., fts_query or vector_query that don't match military service laws)
- Fallen back to raw query (if LLM call failed), which means the raw user query "پسری که مشمول باشه..." is used directly — this is a conversational question, not a keyword query, so it would likely fail FTS matching

**Key code:** [`route_question()`](src/backend/conversations/question_router.py:205)

### Hypothesis B: Query Formulation / HyDE Failure
The question router generates per-hub `vector_query` (HyDE hypothetical answers). If the HyDE text doesn't semantically match the military service law chunks, vector search will fail. For example:
- The HyDE might focus on "فرار از خدمت" (desertion) but the actual law text uses different terminology like "غیبت" (absence), "تخلف" (violation), etc.
- The HyDE might be too short or generic

**Key code:** [`multi_hub_search()`](src/backend/conversations/global_rag_service.py:88)

### Hypothesis C: FTS Keyword Mismatch
The `fts_query` generated by the question router is used for PostgreSQL `websearch` FTS, which ANDs all terms. If any term doesn't exist in the chunk's `search_vector`, the chunk won't be found. This was already demonstrated in `diag_output3.txt` for a different query.

For example, if the fts_query is "فرار از خدمت مشمول" but the actual law text uses "غیبت" or "تخلف از خدمت", the FTS will return 0 results.

**Key code:** [`_keyword_search_by_hub()`](src/backend/documents/services/search_service.py:1095)

### Hypothesis D: Vector Search Embedding Mismatch
The `vector_query` (HyDE) is embedded using `bge-m3` via Ollama. If the HyDE text doesn't semantically align with the military service law chunks, the cosine similarity will be low. The `min_score` defaults to 0.0, so this shouldn't filter results, but low similarity means chunks might not rank high enough in RRF fusion.

**Key code:** [`_vector_search_by_hub()`](src/backend/documents/services/search_service.py:1027)

### Hypothesis E: Data Not Properly Imported
The military service laws might:
- Not exist in the database at all
- Exist but with wrong `hub_type` (e.g., stored as `user_upload` instead of `reference_law`)
- Exist but without embeddings (embedding is null)
- Exist but without `search_vector` (FTS not populated)
- Exist but with wrong `hub_type` on the chunks (e.g., `hub_type` is null or wrong)

**Key code:** [`import_chunked_data.py`](src/backend/documents/management/commands/import_chunked_data.py:151), [`import_reference_laws.py`](src/backend/documents/management/commands/import_reference_laws.py:89)

### Hypothesis F: Top-K Too Low
The `top_k_per_hub` is set to 5 (in [`views.py:373`](src/backend/conversations/views.py:373)). If the relevant chunks rank below position 5 in any search method, they won't be retrieved. With RRF fusion, a chunk needs to appear in at least one search method's top results to be considered.

### Hypothesis G: Context Token Budget Truncation
The `build_global_context()` function truncates to `RAG_CONTEXT_TOKEN_BUDGET` chars. If the budget is too small, relevant chunks might be truncated out of the context before being sent to the LLM.

**Key code:** [`build_global_context()`](src/backend/conversations/global_rag_service.py:179)

## Diagnostic Steps

### Step 1: Check What Reference Law Data Exists in the Database

Run a Django diagnostic script that:
1. Lists all `Document` records with `document_type='reference_law'`
2. Shows their `hub_type`, `title`, `status`, `total_chunks`
3. For each, shows chunk count, embedded count, search_vector count
4. Specifically searches for military-service-related documents (e.g., "نیروهای مسلح", "خدمت", "سربازی", "مجازات", "فرار")
5. Shows the `hub_type` distribution of chunks

**Script:** `scripts/diag_step2_check_data.py`

### Step 2: Trace the Question Router for the Specific Query

Create a script that:
1. Calls `route_question()` with the exact user query
2. Logs the full `RouterResult` including:
   - Which hubs were activated
   - The `fts_query` and `vector_query` for each hub
   - The reasoning from the LLM
3. Saves this output for analysis

**Script:** `scripts/diag_step3_trace_router.py`

### Step 3: Trace the Multi-Hub Search

Create a script that:
1. Takes the `RouterResult` from Step 2
2. For each active hub, calls `multi_hub_search()` 
3. Logs:
   - The embedding generated from `vector_query`
   - The number of results from vector, keyword, and trigram searches
   - The top 5 chunks from each hub with their scores
   - Whether any military-service-related chunks were found
4. Saves this output for analysis

**Script:** `scripts/diag_step4_trace_search.py`

### Step 4: Direct Search with Manual Queries

Create a script that:
1. Tests various search queries directly against the database:
   - "فرار از خدمت"
   - "مشمول غایب"
   - "قانون مجازات جرایم نیروهای مسلح"
   - "تخلف از خدمت سربازی"
   - "غیبت سربازی"
2. For each query, runs:
   - Vector search (embedding the query)
   - Keyword search (FTS websearch)
   - Trigram search
   - Full hybrid search
3. Shows which chunks are found and from which documents/hubs

**Script:** `scripts/diag_step5_direct_search.py`

### Step 5: Check the Conversation's Stored Hub Metadata

The system stores `hub_metadata` in the `Message` model (see [`migrations/0002_message_hub_metadata.py`](src/backend/conversations/migrations/0002_message_hub_metadata.py)). Create a script that:
1. Fetches the specific conversation `f8c57557-9773-46ff-a688-ab2f0100908f`
2. Gets the assistant message and its `hub_metadata`
3. Shows what was actually retrieved per hub

**Script:** `scripts/diag_step6_check_conversation.py`

## Expected Outputs

For each step, the diagnostic scripts should output:
- **PASS**: The step completed successfully and the data looks correct
- **FAIL**: The step revealed a problem, with specific details
- **INFO**: Additional context that helps understand the behavior

## Root Cause Analysis

After running all diagnostic steps, we should be able to answer:

1. **Does the relevant data exist?** (Step 1)
   - If NO → Data import issue
   - If YES → Continue to Step 2

2. **Did the question router activate the right hubs with good queries?** (Step 2)
   - If NO → Question router issue (prompt, LLM, or parsing)
   - If YES → Continue to Step 3

3. **Did the search retrieve the relevant chunks?** (Step 3)
   - If NO → Search service issue (embedding, FTS, or RRF)
   - If YES → Continue to Step 4

4. **Did the partial answer generation or synthesis drop the relevant info?** (Step 5)
   - If NO → LLM response generation issue
   - If YES → Context building or token budget issue

## Implementation

The diagnostic scripts should be implemented as Django management commands or standalone scripts that can be run via:
```bash
docker-compose exec backend python scripts/diag_step2_check_data.py
```
