# WIP Context — Fix: Duplicate GIN Index Migration Error

## Status: ✅ FIX APPLIED

## What was done

Fixed 75 test errors in `conversations/tests/` caused by:

```
django.db.utils.ProgrammingError: relation "chunk_search_vector_gin" already exists
```

### Root Cause

The GIN index `chunk_search_vector_gin` on `document_chunks.search_vector` was being created **three times** across the migration chain:

1. **Migration `0006`** — `RunSQL` with `CREATE INDEX IF NOT EXISTS` ✅
2. **Migration `0014`** — `RunSQL` with `CREATE INDEX IF NOT EXISTS` ✅
3. **Migration `0017`** — Django `AddIndex` operation (no `IF NOT EXISTS`) ❌

Migration `0017` was auto-generated by `makemigrations` after the `GinIndex` was added to the model's `Meta.indexes`. Django's `AddIndex` translates to `CREATE INDEX ...` **without** `IF NOT EXISTS`, so when the test runner runs all migrations sequentially on a fresh test database, `0017` crashes because the index already exists from `0006`.

### Fix Applied

**File:** [`src/backend/documents/migrations/0017_documentchunk_chunk_search_vector_gin.py`](src/backend/documents/migrations/0017_documentchunk_chunk_search_vector_gin.py)

Replaced the `AddIndex` operation with `RunSQL` using `CREATE INDEX IF NOT EXISTS`, matching the safe pattern used in migrations `0006` and `0014`.

### Files Modified

- `src/backend/documents/migrations/0017_documentchunk_chunk_search_vector_gin.py` — Changed `AddIndex` → `RunSQL` with `IF NOT EXISTS`
